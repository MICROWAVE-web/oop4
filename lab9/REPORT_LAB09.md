## 1. Краткое описание назначения реализуемых паттернов

### Factory Method (Фабричный метод)

Паттерн выделяет создание конкретного продукта в отдельный метод-создатель.  
В контексте ассистента ЛР4 это позволяет создавать семейство `MusicService` через набор специализированных создателей (`SpotifyCreator`, `VKCreator`, `LocalFileCreator`) вместо прямых `if/else` в composition root.

### Abstract Factory (Абстрактная фабрика)

Паттерн задает интерфейс создания **согласованного набора** зависимостей.  
Для ассистента проектируются фабрики профилей (`OfflineFactory`, `SmartFactory`, `CarFactory`), каждая возвращает совместимый комплект: `MusicService`, `InfoService`, `Player`, `ASRModule`, `NLUModule`, `TTSModule`, `VoiceInterface`.

### Singleton (Одиночка)

Паттерн обеспечивает существование единственного экземпляра и глобальной точки доступа.  
В проекте используется для `AssistantConfig`: один объект конфигурации (режим, лимиты пула, параметры прототипов), который читают фабрики и сборщик.

### Prototype (Прототип)

Паттерн позволяет создавать новые объекты через клонирование подготовленных экземпляров.  
В ассистенте это удобно для преднастроенных профилей модулей (например, `NLUPrototype`, `TTSPrototype`), где клон наследует базовые настройки и может донастраиваться под конкретный режим.

### Object Pool (Пул объектов)

Паттерн переиспользует ограниченные/дорогие объекты через выдачу и возврат в пул.  
В архитектуре ЛР09 пул проектируется для `Player` или `TTSSession`: `acquire()` выдает свободный объект, `release()` возвращает его обратно, снижая частоту создания объектов и стабилизируя нагрузку.

## 3. Краткое описание назначения классов в системе


| Класс / интерфейс           | Назначение                                                                                                                       |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `AssistantConfig`           | `Singleton`: хранит единую конфигурацию системы и предоставляет `get_instance()`.                                                |
| `AssistantComponentFactory` | Интерфейс `Abstract Factory` для создания полного комплекта компонентов ассистента.                                              |
| `OfflineFactory`            | Конкретная фабрика офлайн-профиля, формирует согласованный набор локальных/простых сервисов.                                     |
| `SmartFactory`              | Конкретная фабрика облачного профиля, формирует набор расширенных сервисов.                                                      |
| `CarFactory`                | Конкретная фабрика автомобильного профиля, балансирует онлайн и офлайн-компоненты.                                               |
| `MusicServiceCreator`       | Базовый создатель `Factory Method` для семейства `MusicService`.                                                                 |
| `SpotifyCreator`            | Конкретный `Factory Method`, создает `SpotifyMusicService`.                                                                      |
| `VKCreator`                 | Конкретный `Factory Method`, создает `VKMusicService`.                                                                           |
| `LocalFileCreator`          | Конкретный `Factory Method`, создает `LocalFileMusicService`.                                                                    |
| `ModulePrototype`           | Базовый интерфейс `Prototype` с методом `clone()`.                                                                               |
| `NLUPrototype`              | Прототип профиля NLU с предустановленными параметрами распознавания намерений.                                                   |
| `TTSPrototype`              | Прототип профиля TTS с предустановленными параметрами синтеза голоса.                                                            |
| `CloneRegistry`             | Реестр прототипов: хранит шаблоны и отдает клоны по ключу.                                                                       |
| `PooledResource`            | Контракт объекта, который может использоваться в пуле (`reset_state()`).                                                         |
| `PlayerSession`             | Пуллируемый объект воспроизведения/сессии вывода звука.                                                                          |
| `ObjectPool`                | Управляет коллекцией свободных/занятых объектов, предоставляет `acquire()` и `release()`.                                        |
| `AssistantAssembler`        | Компоновщик, который через `Abstract Factory`, `Prototype` и `Object Pool` подготавливает зависимости и создает `DialogManager`. |
| `DialogManager`             | Существующий центральный координатор ЛР4; в ЛР09 остается клиентом собранных зависимостей.                                       |

## Соответствие сущностей ЛР4 и ролей ЛР09

- Интерфейсы из `interfaces.py` используются как целевые абстрактные продукты для `Abstract Factory`.
- Реализации сервисов из `music_services.py`, `info_services.py`, `player.py`, `asr.py`, `nlu.py`, `tts.py`, `voice_interfaces.py` выступают конкретными продуктами фабрик.
- `main.py` ЛР4 с ручной сборкой конфигураций концептуально заменяется проектируемым `AssistantAssembler`.
- `DialogManager` сохраняется без изменения ответственности и получает зависимости через спроектированные порождающие механизмы.

## Итог

В ЛР09 построена целостная проектная модель, которая:

- сохраняет архитектурное ядро ЛР4 (`DialogManager` + интерфейсы),
- выносит создание и настройку объектов в порождающие паттерны,
- повышает расширяемость за счет единых фабрик профилей, прототипов модулей и переиспользования ресурсов через пул.
