## 1. Краткое описание назначения реализуемых паттернов

### State (Состояние)

Поведение объекта зависит от его внутреннего состояния, которое может меняться во время работы. Паттерн помещает каждое состояние в отдельный класс и делегирует ему запросы текущего объекта-контекста. Это устраняет разветвлённые `if/else` по полю «текущий режим».

В системе ЛР4 одной и той же команде (например, «включи музыку») должен соответствовать разный отклик в разных режимах: в покое — запустить трек, во время воспроизведения — сообщить, что трек уже играет, в режиме «не беспокоить» — проигнорировать. Это естественный сценарий для State.

### Memento (Хранитель)

Позволяет зафиксировать внутреннее состояние объекта во внешнем носителе, не нарушая инкапсуляции, и впоследствии восстановить его. Подходит для механизмов отката/«запомнить-восстановить».

В ассистенте мы фиксируем «настройки сеанса» (громкость, текущий трек, имя текущего состояния), чтобы по команде пользователя «запомни настройки» сохранить снимок и по команде «восстанови настройки» вернуться к нему.

### Observer (Наблюдатель)

Устанавливает связь «один ко многим»: при изменении состояния субъекта все подписавшиеся наблюдатели получают уведомление. Это разрывает жёсткую связь между источником событий и заинтересованными в них компонентами.

В ассистенте субъектом становится диалоговый менеджер, а подписчики — независимые модули, реагирующие на события: журналирование, сбор статистики, имитация экрана колонки. Любой новый подписчик добавляется без изменения логики менеджера.

## 3. Краткое описание назначения классов в системе

### Контекст

- `DialogManager` — исходный диалоговый менеджер из ЛР4 (без изменений).
- `StatefulDialogManager` — наследник `DialogManager`. Хранит ссылку на текущее состояние, хранитель снимков и субъект событий. Делегирует обработку намерения текущему состоянию и публикует события подписчикам.

### State

- `AssistantState` — абстрактный базовый класс состояния. Определяет метод `handle(ctx, intent, data)` и `name()`.
- `IdleState` — режим ожидания: реагирует на все основные намерения (музыка, новости, информация, громкость), переключает контекст в `PlayingState` при запуске трека.
- `PlayingState` — режим воспроизведения: отдельно реагирует на повторное «включи музыку» («уже играет»), команду «стоп» (возвращает в `IdleState`).
- `MutedState` — режим «не беспокоить»: игнорирует пользовательские команды кроме «вернись» (возврат в `IdleState`) и «выключись».

### Memento

- `SettingsMemento` — иммутабельный снимок настроек: уровень громкости, идентификатор текущего трека, имя текущего состояния.
- `SettingsCaretaker` — хранитель: держит стек снимков, поддерживает операции `save(memento)` и `restore()`. Не имеет доступа к внутренним полям контекста — работает только с готовыми снимками.

### Observer

- `EventSubject` — субъект: ведёт список подписчиков, рассылает им события `(name, payload)`.
- `EventObserver` — абстрактный наблюдатель с методом `update(name, payload)`.
- `LoggerObserver` — пишет события в текстовый лог (`run.log`).
- `StatisticsObserver` — считает счётчики (число запущенных треков, число изменений громкости, число снимков).
- `ScreenObserver` — имитация экрана колонки: печатает «текущий экран» при каждом значимом событии.

## 4. Постановка задачи конфигурирования системы

В ЛР4 диалоговый менеджер собирался напрямую в `main.py`, в ЛР11 этот процесс был перенесён в Builder + Director. В ЛР13 предполагается:

1. **Сохранить совместимость с архитектурой ЛР4 и сборкой ЛР11.**  
   `StatefulDialogManager` собирается на основе уже готового набора зависимостей (`AssistantAssembly` из `lab11/builder.py`): музыкальный сервис, плеер, ASR/NLU/TTS и голосовой интерфейс остаются прежними.

2. **Конфигурирование начального состояния.**  
   Клиентский код задаёт стартовое состояние (по умолчанию `IdleState`). Переключения между состояниями выполняются самими состояниями через `context.set_state(...)`. Это вариант 2 из лекционного материала: «каждое состояние программируется индивидуально».

3. **Конфигурирование набора подписчиков.**  
   Клиент создаёт нужные наблюдатели и регистрирует их через `subject.attach(...)`. Состав подписчиков варьируется без изменения кода `StatefulDialogManager` — это и есть преимущество Observer.

4. **Конфигурирование хранителя снимков.**  
   `SettingsCaretaker` создаётся в одном экземпляре на сеанс. Команды пользователя «запомни настройки» / «восстанови настройки» транслируются NLU в новые intents (`snapshot` / `restore`) и обрабатываются менеджером.

5. **Расширение NLU.**  
   В ЛР13 потребуется подкласс `RuleBasedNLU` (`ExtendedRuleBasedNLU`), который дополнительно распознаёт ключевые слова: «запомни», «восстанови», «не беспокой», «вернись».

Итог: ЛР13 не переписывает архитектуру ЛР4/ЛР11, а наращивает её за счёт трёх паттернов поведения. Существующие профили (offline/smart/car) продолжают работать; меняется только тип итогового менеджера и подключение наблюдателей.
