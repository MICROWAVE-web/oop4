---

## 1. Краткое описание назначения реализуемого паттерна

**Builder (Строитель)** позволяет отделить процесс пошаговой сборки сложного объекта от его итогового представления.

В контексте системы из ЛР4 сложным объектом является **настроенный `DialogManager`** с набором зависимостей:
- `MusicService`
- `InfoService`
- `Player`
- `ASRModule`
- `NLUModule`
- `TTSModule`
- `VoiceInterface`

В ЛР11 сборка этой конфигурации выполняется не напрямую в `main.py`, а через:
- `AssistantBuilder` (набор шагов сборки),
- конкретные строители (`OfflineAssistantBuilder`, `SmartAssistantBuilder`, `CarAssistantBuilder`),
- `AssistantDirector` (фиксирует порядок шагов).

## 3. Краткое описание назначения классов в системе

- `AssistantAssembly` — контейнер собранных зависимостей; превращает их в готовый `DialogManager`.
- `AssistantBuilder` — абстрактный строитель, задающий шаги сборки конфигурации ассистента.
- `OfflineAssistantBuilder` — строит офлайн-профиль (Proxy + Wikipedia + BasicTTS + RuleBasedNLU и т.д.).
- `SmartAssistantBuilder` — строит online/smart-профиль (Google, NeuralTTS, вывод на bluetooth и т.д.).
- `CarAssistantBuilder` — строит автомобильный профиль (VK + Yandex + др. настройки).
- `AssistantDirector` — управляет порядком шагов сборки и возвращает результат.
- `ScriptedVoiceInterface` — тестовый голосовой интерфейс для воспроизводимых логов в демонстрации.

## 4. Постановка задачи конфигурирования системы

В ЛР4 конфигурации формировались вручную в `main.py` через прямое создание объектов.

Задача ЛР11:
1. Перенести конфигурирование в паттерн **Builder**, чтобы процесс сборки был единообразным и расширяемым.
2. Поддержать несколько профилей конфигурации (offline/smart/car) без дублирования общего алгоритма сборки.
3. Сохранить совместимость с архитектурой ЛР4 (использовать существующие сервисы через импорт из корня проекта).

## Сравнительный анализ порождения объектов (ЛР4 vs ЛР11)

- **Подход ЛР4:** ручная сборка зависимостей в `main.py`.
  - Плюс: просто для старта.
  - Минус: при росте числа профилей растёт дублирование.

- **Подход ЛР11:** Builder + Director.
  - Плюс: единый сценарий сборки, профиль задаётся конкретным строителем.
  - Плюс: проще добавлять новые профили без изменения клиентского кода.
  - Минус: больше служебных классов, чем при «прямом» создании.

Итог: для системы с несколькими конфигурациями и повторным использованием ресурсов вариант ЛР11 более масштабируем и удобен в сопровождении.
