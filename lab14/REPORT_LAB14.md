## 1. Краткое описание назначения реализуемых паттернов

### Command (Команда)

Преобразует запрос на выполнение операции в самостоятельный объект. Это разрывает связь между объектом, инициирующим операцию (Invoker), и объектом, который умеет её выполнять (Receiver). Команды легко передавать, ставить в очередь, протоколировать и накапливать в истории.

В ЛР4 диалоговый менеджер сам решал, что делать, по таблице обработчиков `handle_intent`. В ЛР15 каждое распознанное намерение пользователя будет «упаковано» в объект-команду со своим Receiver-сервисом и единым методом `execute()`.

### Chain of Responsibility (Цепочка обязанностей)

Связывает объекты-обработчики в цепочку: запрос идёт по ней, пока кто-то не возьмётся обработать (или пока цепочка не закончится). Каждое звено отвечает за одну обязанность и не знает о других; цепочку можно динамически перестраивать.

В ассистенте цепочка применяется до распознавания намерения — для предобработки сырого текста: отброс пустых строк, нормализация регистра и пробелов, журналирование. Любое звено может остановить запрос, вернув «пусто», и тогда NLU не вызывается.

### Visitor (Посетитель)

Позволяет добавить новую операцию над набором объектов (структурой), не меняя классы самих объектов. Операции выносятся в отдельный класс-посетитель; элементы структуры лишь принимают посетителя через `accept(v)` и сообщают ему свой конкретный тип.

В ассистенте структурой становится история выполненных команд. Поверх неё можно строить любую аналитику (статистика, журнал сеанса, список нераспознанных запросов) без изменения самих команд.

## 3. Краткое описание назначения классов в системе

### Контекст

- `DialogManager` — исходный диалоговый менеджер из ЛР4 (без изменений).
- `CommandDialogManager` — наследник `DialogManager`. Хранит ссылку на первое звено цепочки предобработчиков, фабрику команд и инвокер. Переопределяет `run()`: пропускает текст пользователя через цепочку, для прошедшего текста вызывает `NLU.parse`, по `(intent, data)` собирает команду и отдаёт её инвокеру.

### Command

- `Command` — абстрактная команда: методы `execute()` и `accept(visitor)`.
- `PlayMusicCommand` — запуск трека через `MusicService` (Receiver).
- `SetVolumeCommand` — установка громкости через `Player` (Receiver).
- `GetInfoCommand` — справочный запрос к `InfoService`.
- `GetNewsCommand` — запрос новостей у `InfoService`.
- `ExitCommand` — завершение сеанса через `DialogManager.stop()`.
- `UnknownCommand` — фоллбэк для нераспознанных намерений; хранит исходный текст для последующего анализа визиторами.
- `CommandFactory` — фабрика, которая по `(intent, data)` собирает нужную конкретную команду; зависимости (Receiver-сервисы) передаются в её конструктор один раз.
- `CommandInvoker` — выполняет команды и сохраняет их в публичный список `history`. Именно история становится «структурой», по которой потом ходит Visitor.

### Chain of Responsibility

- `TextHandler` — абстрактный обработчик с методами `set_next(handler)` и `handle(text) -> Optional[str]`. Возвращает `None`, если запрос отброшен, иначе — преобразованный текст для следующего звена.
- `StripHandler` — удаляет ведущие/хвостовые пробелы.
- `EmptyTextHandler` — отсекает пустые строки (после `Strip`); прерывает цепочку.
- `LowercaseHandler` — приводит текст к нижнему регистру для устойчивости NLU.
- `LoggingHandler` — финальное звено: пишет в консоль, что текст прошёл фильтры, и возвращает его без изменений.

### Visitor

- `CommandVisitor` — абстрактный посетитель: по одному методу `visit_xxx` на каждый тип команды.
- `CommandStatisticsVisitor` — считает количество команд каждого типа за сеанс; формирует строковый отчёт `report()`.
- `CommandLogVisitor` — собирает человекочитаемый журнал из истории команд (что выполнялось и с какими данными).
- `UnknownCommandsVisitor` — извлекает из истории все нераспознанные запросы (исходный текст пользователя), полезно для анализа покрытия NLU.

### Receiver-сервисы (используются как есть из ЛР4)

- `MusicService` (`ProxyMusicService`/`SpotifyMusicService` и др.) — поиск и воспроизведение треков.
- `Player` (`BasicPlayer`) — управление громкостью.
- `InfoService` (`WikipediaInfoService` и др.) — справочные ответы и новости.
- `NLUModule` (`RuleBasedNLU`) — внешний поставщик `(intent, data)` для фабрики команд.

## 4. Постановка задачи конфигурирования системы

В ЛР4 диалоговый менеджер собирался напрямую в `main.py` через Dependency Injection. В ЛР15 эта схема сохраняется, но дополняется тремя новыми «сборочными узлами» — цепочкой обработчиков, фабрикой команд и набором визиторов. Никакого Builder из ЛР11 не используется: конфигурация остаётся явной и видимой в одном файле.

Шаги конфигурирования (план для ЛР15):

1. **Базовые сервисы ЛР4 без изменений.**  
   Клиент создаёт стандартный набор: `ProxyMusicService(SpotifyMusicService(), offline=True)`, `WikipediaInfoService`, `BasicPlayer`, `SimpleASR`, `RuleBasedNLU`, `BasicTTS`, тестовый `ScriptedVoiceInterface`. Это полностью совпадает с офлайн-конфигурацией исходного `main.py`.

2. **Сборка цепочки предобработчиков (CoR).**  
   Клиент создаёт звенья `StripHandler`, `EmptyTextHandler`, `LowercaseHandler`, `LoggingHandler` и связывает их через `set_next(...)`. Состав и порядок задаются клиентом и могут меняться без правки `CommandDialogManager`. Цепочка — это «один из преимуществ Chain of Responsibility»: возможность динамической настройки.

3. **Сборка фабрики команд (Command).**  
   Клиент создаёт `CommandFactory`, передавая ей Receiver-сервисы (`MusicService`, `Player`, `InfoService`) и ссылку на самого менеджера (для `ExitCommand`). После этого фабрика умеет по `(intent, data)` сконструировать любую известную команду. Добавление нового намерения сводится к добавлению нового конкретного `Command` и одной ветки в `create(...)`.

4. **Создание инвокера.**  
   Один экземпляр `CommandInvoker` на сеанс. Его публичная история станет структурой, по которой пройдут визиторы.

5. **Создание расширенного менеджера.**  
   `CommandDialogManager` принимает все зависимости ЛР4 + первое звено цепочки + фабрику + инвокер. В `run()` он:  
   - получает текст из ASR;  
   - пропускает его через цепочку (`chain.handle(text)`); если пришло `None` — итерация пропускается;  
   - вызывает `nlu.parse(text)` и фабрику; полученную команду передаёт инвокеру;  
   - после завершения цикла история команд готова к обходу.

6. **Регистрация визиторов (Visitor).**  
   Клиент создаёт нужный набор визиторов (`CommandStatisticsVisitor`, `CommandLogVisitor`, `UnknownCommandsVisitor`) и после `manager.run()` поочерёдно прогоняет их по `invoker.history`: для каждой команды вызывается `cmd.accept(visitor)`. Состав аналитики определяется только клиентом — внутренности команд не меняются.

Итог: ЛР15 не переписывает архитектуру ЛР4, а наращивает её за счёт трёх паттернов поведения. Существующая офлайн-конфигурация продолжает работать; меняется только тип диалогового менеджера и появляются три новых модуля (`commands`, `preprocessing`, `visitors`), которые подключаются явно в `main.py`.
